Agora que falei o que quem odeia a linguagem adora escutar, acho que dá pra prosseguir normalmente e conversar como adultos, sobre um problema que eu senti e que você também pode sentir também com a linguagem.

Fato é, eu gosto muito de Java. Quando tive meu primeiro contato com programação com 15 anos, minhas primeiras aulas de algoritmos e linguagem de programação foram nessa linguagem, ao contrario do padrão C da época, e Python atualmente. Posteriormente assumi meu primeiro estágio desenvolvendo em SpringBoot com Angular 8 e desde então Java foi quem me abriu portas no mercado de trabalho e me fez estar onde estou atualmente.

Porém uma coisa que percebia enquanto trabalhava é que o Java sempre foi engessado em relação a dependências e código distribuido. O mais padrão em um projeto empresarial com Java é utilizar as funções do Kit de Desenvolvimento Java (JDK) e, caso não tenha jeito, usar gerenciadores de dependencias como Maven ou Gradle, onde as dependencias que são listadas em um arquivo de configuração (pom.xml/build.gradle) são baixadas, compiladas e ficam disponíveis ao codigo fonte.

A princípio não é um sistema ruim, a biblioteca padrão do Java é bem robusta e dificilmente falta alguma coisa lá, precisando de algumas poucas coisas que geralmente um Guava vai possuir. Entretando, não dá pra negar que é um mecanismo bastante burocrático.

Atualmente estou trabalhando bastante com C fora do trabalho, como um hobby. E, uma das coisas que me chama atenção é justamente como as coisas funcionam naquela linguagem. Por mais difícil que alguns aspectos da linguagem parecem ser, C é uma linguagem bem direta nos seus propósitos: Ao adicionar um #include<stdio.h> o código do header stdio.h é colado no arquivo que inclui ele.

E justamente esse funcionamento dos headers em C que possibilitou a existencia das dependencias "header-only". Estas dependências são arquivos de cabeçalho com todo o código/estruturas de dados necessárias para que a biblioteca funcione independentemente, sendo necessário apenas baixar o cabeçalho, colar no seu código e usar.

Portanto, pensando nesse principio de dependências "header-only" que eu criei o projeto "Jstd". A ideia dele é ser justamente uma coletânea de estruturas de dados e algoritmos que são autocontidos, sendo necessário apenas copiar, colar no seu projeto Java e utilizar.

Outra vantagem desta abordagem é justamente o fato do código fonte estar com VOCÊ. Isso da a liberdade de justamente poder moldar o código ao seu bem-querer.

Quer alterar o funcionamento de um algoritmo? Tranquilo.

Quer extender de uma estrutura de dados e mudar os modificadores de acesso de private para protected? No problem, vá em frente.

Atualmente o código fonte está disponível no Github. Sinta-se a vontade para contribuir, ainda mais agora que o projeto está no início.
